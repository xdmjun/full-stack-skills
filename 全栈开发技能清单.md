## 工作流使用方法

### 什么是工作流？

工作流是将多个技能按特定顺序组合使用，以完成某个开发阶段或完整项目。工作流用 `→` 箭头连接多个技能，表示执行顺序。

### 工作流的基本用法

#### 1. 安装所需技能

首先安装工作流中涉及的所有技能：

```bash
# 示例：安装需求分析工作流的技能
npx skills add obra/superpowers@brainstorming
npx skills add jwynia/agent-skills@requirements-analysis
npx skills add obra/superpowers@writing-plans
```

#### 2. 按顺序使用技能

在 AI 助手中按工作流顺序依次调用各技能：

```
用户：使用 brainstorming 技能帮我分析这个产品想法...
AI：[使用 brainstorming 技能进行苏格拉底式提问]

用户：现在使用 requirements-analysis 技能整理需求文档
AI：[使用 requirements-analysis 技能生成结构化文档]

用户：使用 writing-plans 技能制定实施计划
AI：[使用 writing-plans 技能分解任务]
```

#### 3. 灵活组合

可根据项目实际情况：

- 选择部分工作流（如只做需求分析）
- 调整顺序（如跳过某些步骤）
- 并行执行（如前后端同时开发）

### 工作流使用技巧

| 技巧     | 说明                     |
| -------- | ------------------------ |
| 先装后用 | 确保技能已安装再调用     |
| 循序渐进 | 按箭头顺序逐步执行       |
| 检查输出 | 每步完成后确认结果再继续 |
| 迭代优化 | 可回退到前面步骤重新执行 |
| 保存记录 | 将各阶段输出保存为文档   |

### 常见工作流场景

| 场景 | 推荐工作流 | 说明 |
| ---------- | ----------------------------------------------------------------------- | ---- |
| 新项目启动 | `brainstorming → requirements-analysis → writing-plans` | 从想法到可执行计划 |
| 功能开发 | `frontend-blueprint → coding → test-driven-development` | 设计驱动开发 |
| Bug 修复 | `systematic-debugging → verification-before-completion` | 根因分析确保修复 |
| 代码审查 | `requesting-code-review → receiving-code-review` | 双向审查流程 |
| 项目部署 | `using-git-worktrees → ci-cd → deploy → finishing-a-development-branch` | 安全部署流程 |
| 数据库设计 | `database-schema-design → database-schema-designer → api-design` | 从模型到接口 |
| API 开发 | `api-design-framework → api-design → api-design-patterns` | 规范化 API 设计 |
| 前端重构 | `refactoring-surgeon → lint-fix → test-driven-development` | 安全重构流程 |
| 团队协作 | `writing-plans → dispatching-parallel-agents → executing-plans` | 并行任务分配 |
| AI 应用开发 | `prompt-engineering → langchain → rag-implementation` | LLM 应用构建 |
| 性能优化 | `best-practices → systematic-debugging → verification-before-completion` | 定位并解决性能问题 |
| 技术债务清理 | `code-review → refactoring-surgeon → lint-fix` | 识别并清理债务 |
